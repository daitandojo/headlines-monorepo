Memory Systems Are Solving the Wrong Problem

Vector databases are fast. Memory systems are accurate. Neither one can tell you who owns what, who reports to whom, or which facts contradict each other. Here's why structured fact intelligence is a missing category — and what we built instead.

By an anonymous builder · 15 min read

The Problem

In 2025, the AI memory market exploded. Mem0 raised $24M. Pinecone hit 4,000 paying customers. Zep published a paper showing 94.8% on the Deep Memory Retrieval benchmark. Everyone was racing to build the best memory for AI agents.

But there's a class of problems these systems cannot touch:

"Show me all UK properties owned by Danish business families, with source citations and confidence scores."

A vector database returns similar documents. A memory system returns relevant chat history. Neither can answer this question because neither extracts entities, builds relationships, tracks provenance, or detects contradictions.

This isn't a memory problem. It's a knowledge graph problem. And the existing tools — from Qdrant at 3ms vector search to Mem0 at 93% LongMemEval accuracy — were not designed for it.

The Landscape

Let's be honest about what exists and what it actually does.

Vector Databases (Qdrant, Pinecone, Weaviate, pgvector)

These store embeddings and return nearest neighbors. They are extremely good at this one thing: Qdrant achieves 3ms P50 latency at 1M vectors. pgvector costs nothing if you're already on Postgres.

What they cannot do:

  • Extract entities from unstructured text
  • Build relationships between facts
  • Track which source confirmed a fact
  • Detect that "net worth: $4.6B" contradicts "net worth: $5.2B"
  • Answer "who owns what?" as a structured query

Vector databases are a component, not a solution. Teams building knowledge applications must glue together embedding models, extraction pipelines, graph databases, and search indexes — a fragile stack that requires engineering teams to maintain.

Memory Systems (Mem0, MemGPT, Zep)

These solve a different problem: conversational memory for AI agents. Mem0's v2 algorithm achieves 93.4% on LongMemEval with 6,800 tokens per query — genuinely impressive for chat recall.

What they cannot do:

  • Extract facts about multiple entities from a single document
  • Build a knowledge graph with typed entities and relationships
  • Distinguish "Troels Holch Povlsen founded Bestseller" from "Anders Holch Povlsen inherited Bestseller"
  • Attribute facts to specific source documents
  • Handle the "data room" use case at all

The Approach

Rather than optimizing a single metric (latency, accuracy, token efficiency), we optimized for a single question: can this system turn a stack of documents into a queryable knowledge graph?

The answer required building what existing systems leave as exercises for the reader:

  1. Automatic entity extraction — Every document is processed by an LLM (deepseek-v4-flash) that extracts self-contained atomic beliefs with explicit subject, predicate, object, and entity types.
  2. Multi-entity separation — A single article mentioning 3 people and 2 companies produces 5 separate entity profiles, not one document summary.
  3. Knowledge graph population — Every belief automatically creates or updates graph entities and relationships. No manual schema design. The graph grows organically.
  4. Hybrid search — Vector similarity (70%) combined with full-text keyword search (30%) through pgvector.
  5. Graph query API — Entity lookup, relationship browsing, and BFS graph traversal are first-class endpoints.
  6. Source provenance — Every belief tracks which sources confirmed it, with reinforcement on repeat confirmation.
  7. Contradiction detection — Conflicting facts are preserved and flagged, not silently overwritten.
3 × 3 factorial study (arXiv:2603.02473): Hybrid reranking achieved 77.2% accuracy vs BM25's 57.1% — a 20-point gap from retrieval method alone. Retrieval failure was the dominant error mode (11-46% of all questions).

Architecture

POST /v1/memories → deepseek-v4-flash splits into atomic beliefs ↓ text-embedding-3-small generates 1536-dim vector per belief ↓ pgvector IVFFlat index stores for cosine similarity search ↓ graph_entities table: 25 columns (id, type, name, mentions ...) ↓ graph_relationships table: 20 columns (source, target, predicate, ...) ↓ entity_aliases table: canonical name → alias mapping ↓ belief_conflicts: contradictions flagged, not overwritten GET /v1/graph/entities → list/search with type filter GET /v1/graph/entities/:name → profile + all beliefs GET /v1/graph/entities/:name/relationships → incoming + outgoing POST /v1/graph/traverse → BFS graph traversal POST /v1/memories/search → hybrid (0.7 vector + 0.3 keyword)

Cost per article: ~$0.0037 (1,046 input tokens + 8,881 output tokens on deepseek-v4-flash + 34 embeddings on text-embedding-3-small).

Infrastructure: 32MB heap, 122MB RSS — runs on a $10/month VPS with PostgreSQL and pgvector.

Honest assessment: Cogniti is not competitive on pure vector search latency (94ms vs Qdrant's 3ms) because it does dramatically more per query. It is not competitive on LongMemEval accuracy because it wasn't designed for conversational recall. It is the only system that combines entity extraction, graph population, relationship traversal, provenance tracking, and contradiction detection in a single self-hosted stack. No competitor's benchmark measures this capability because no competitor has it.

Business Case

Primary: Data Room Intelligence

A mid-market M&A deal involves reviewing 5,000-50,000 documents. Current practice: junior analysts and contract lawyers read documents, extract entities into Excel, and manually cross-reference. At $500+/hour for a team of 3-5, a single deal costs $50K-250K in document review alone.

Cogniti ingests the data room, extracts every entity, builds the relationship graph, tracks provenance, and surfaces contradictions — in minutes. The output is a queryable knowledge graph that analysts can interrogate in natural language.

Adjacent Verticals

  • Wealth Intelligence — HNWI profiling from public records and news
  • Compliance / AML — PEP screening, ownership structure mapping
  • Legal Discovery — Entity extraction from millions of documents
  • Competitive Intelligence — Company relationship mapping
  • Investigative Journalism — Network mapping from public records
The market gap: Intelligent Document Processing tools extract entities but don't build graphs. Graph databases (Neo4j) require manual schema design. Vector databases retrieve documents but don't extract facts. Memory systems track conversations but don't handle documents. No existing product combines all four.
